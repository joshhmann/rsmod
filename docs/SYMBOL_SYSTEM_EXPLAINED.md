# Symbol System Explained - Why objs/sym Files Cause Problems

## What Are These Files?

### .sym Files (Symbol Tables)
Located in: `rsmod/.data/symbols/`

These are mapping files that connect **human-readable names** to **game IDs**:

```
# obj.sym example
0    mcannonremains
1    mcannontoolkit
2    mcannonball
4587 dragon_scimitar
```

**Purpose:** Instead of writing magic numbers like `obj = 4587` in code, you write `obj = objs.dragon_scimitar`.

### How It Works in Code

```kotlin
// WITHOUT symbols (bad - magic number)
val item = 4587  // What item is this? Who knows!

// WITH symbols (good - readable)
val item = objs.dragon_scimitar  // Clear what this is
```

The symbol system looks up "dragon_scimitar" in `obj.sym`, finds ID `4587`, and uses that.

## The Problems You're Experiencing

### 1. **"Unresolved Reference" Errors**

**What it looks like:**
```
Unresolved reference 'granite_cannonball_233'
```

**Why it happens:**
- Code references a symbol name that doesn't exist in the .sym file
- Someone wrote `objs.granite_cannonball_233` but only `granite_cannonball` exists

**Real example from logs:**
```
ItemExamines.kt:1340:19 Unresolved reference 'granite_cannonball_233'
```

**Fix:** Use the correct symbol name from obj.sym:
```kotlin
// Wrong
obj = find("granite_cannonball_233")

// Right
obj = find("granite_cannonball")
```

---

### 2. **"internalId must not be null" Errors**

**What it looks like:**
```
java.lang.IllegalStateException: internalId must not be null
```

**Why it happens:**
- The `find("name")` call returned null (name not found in .sym)
- But the code tries to use it anyway
- Usually happens at runtime when scripts load

**Common causes:**
1. Symbol name is misspelled
2. Symbol was removed from cache but code still references it
3. Using wiki names instead of rev 233 symbol names

**Example:**
```kotlin
// This fails at runtime if "grimy_guam_leaf" doesn't exist
val herb = find("grimy_guam_leaf")  // null!
// Later use throws "internalId must not be null"
```

**Fix:** Check symbol exists:
```kotlin
// Check obj.sym first - correct name is "grimy_guam"
val herb = find("grimy_guam")
```

---

### 3. **Revision Mismatches (Rev 233 vs Current)**

**The Problem:**
- RSMod targets **Rev 233** (OpenRS2 `runescape/2293`, built `2025-09-10`)
- OSRS Wiki often has **current/live revision** data
- Item/NPC IDs change between revisions!

**Example - Dragon Scimitar:**
```
Rev 184 (old): ID might be 4587
Rev 233: ID is 4587
Current (2025): ID might still be 4587

BUT some items shifted!
```

**Real example from our issues:**
```kotlin
// Wiki says "Granite cannonball" has ID X
// But in rev 233 symbols, it has a different name/ID

// Code uses wiki name (WRONG)
find("granite_cannonball_233")  // Doesn't exist!

// Should use symbol name (RIGHT)
find("granite_cannonball")  // Exists at ID 12842
```

---

### 4. **The "Blank/Null" Problem**

**What it looks like:**
- Scripts fail to load
- NPCs don't spawn
- Items don't appear

**Why it happens:**
```kotlin
// In NpcReferences.kt or similar
val shop_keeper = find("shop_keeper")  // Returns null!
```

If "shop_keeper" isn't in npc.sym, `find()` returns null.
When the script tries to use this NPC, it crashes or skips.

**From BUILD-CRIT-15:**
```
Scripts disabled at startup:
- Lumbridge General Store (shop_keeper internalId null)
- PrayerAltar (prayer altar loc internalId null)
```

**Fix:** Add missing symbols to `.local/` files:
```
# .local/npc.sym
shop_keeper  390  # ID from rev 233 cache
```

## How The Symbol System Actually Works

### Step-by-Step:

1. **Code references symbol:**
   ```kotlin
   val sword = objs.dragon_scimitar
   ```

2. **ObjReferences looks up name:**
   ```kotlin
   // In ObjReferences.kt
   fun find(name: String): ObjType {
       val id = lookupInSymFiles(name)  // Searches .sym files
       return ObjType(id)
   }
   ```

3. **System searches in order:**
   - First: `rsmod/.local/obj.sym` (custom overrides)
   - Second: `rsmod/.data/symbols/obj.sym` (base symbols)

4. **If not found:** Returns null → "internalId must not be null" error

---

## Where .sym Files Come From

### Base Symbols (`rsmod/.data/symbols/`)
- **Source:** Extracted from OSRS cache (rev 233)
- **Generated by:** Cache tools that read the game files
- **Should contain:** All items/NPCs/locations in rev 233

### Local Overrides (`rsmod/.local/`)
- **Purpose:** Add missing symbols or fix incorrect ones
- **Format:** Same as base - `name  id`
- **Priority:** Overrides base symbols

## Common Fixes

### Fix 1: Find Correct Symbol Name
```bash
# Search obj.sym for partial match
grep -i "cannonball" rsmod/.data/symbols/obj.sym

# Output:
# 2     mcannonball
# 12842 granite_cannonball
```

Use `granite_cannonball` NOT `granite_cannonball_233`

### Fix 2: Add Missing Symbol
```bash
# Edit .local/npc.sym
echo "shop_keeper  390" >> rsmod/.local/npc.sym
```

### Fix 3: Check Symbol Exists Before Using
```kotlin
// Defensive coding
val npc = find("shop_keeper") 
    ?: error("shop_keeper not found in symbols!")
```

### Fix 4: Use MCP Tools to Verify
```kotlin
// Use the MCP osrs-cache server
search_objtypes(query: "dragon")
// Returns: dragon_scimitar at ID 4587
```

## Why This Keeps Happening

### 1. **Wiki Data ≠ Rev 233 Data**
- Wiki shows current OSRS
- We need rev 233 (OpenRS2 `runescape/2293`, built 2025-09-10)
- Item names changed, IDs shifted

### 2. **Manual Symbol Entry**
- Developers guessing symbol names
- Not checking .sym files first
- Adding wrong names to code

### 3. **Cache Drift**
- Local cache updates
- Symbols get regenerated
- Names change between generations

### 4. **No Validation**
- No automated check that symbol names exist
- Errors only show at runtime
- Hard to catch before commit

## The Solution Path

### Short Term (Fix Current Issues):
1. ✅ Run `python tools/npc_lookup.py` to get correct rev 233 names
2. ✅ Check `obj.sym`/`npc.sym` before adding new references
3. ✅ Add missing symbols to `.local/` files
4. ✅ Use MCP cache tools to verify IDs

### Long Term (Prevent Future Issues):
1. Build-time symbol validation - fail if reference doesn't exist
2. Auto-generate symbol refs from cache (don't hand-write)
3. Lint rule: No raw `find("string")` without verification
4. Documentation: "Always check .sym files first"

## Quick Reference

### Check if Symbol Exists:
```bash
# Items
grep "item_name" rsmod/.data/symbols/obj.sym

# NPCs
grep "npc_name" rsmod/.data/symbols/npc.sym

# Objects/Locations
grep "loc_name" rsmod/.data/symbols/loc.sym
```

### Add Missing Symbol:
```bash
# Find ID from cache (use MCP tools)
# Then add to local override
echo "symbol_name  ID" >> rsmod/.local/[type].sym
```

### Validate Before Commit:
```bash
# This should return the ID
grep "^symbol_name" rsmod/.data/symbols/*.sym rsmod/.local/*.sym 2>/dev/null
```

## Bottom Line

**The symbol system is a mapping layer between human names and game IDs.**

Problems happen when:
- ❌ Code uses wrong names (wiki vs rev 233)
- ❌ Symbols are missing from .sym files
- ❌ Names are misspelled
- ❌ No validation happens before runtime

**Fix:** Always check `rsmod/.data/symbols/` files for the correct rev 233 names!
